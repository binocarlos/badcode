# animate-slide Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let an operator say *"turn this slide into a video, here's the motion prompt"* and have Claude drive Google Flow image→video, harvest the clip, and weave it into a comic as a scroll-scrubbed `AnimationWidget` — in the BadCode voice.

**Architecture:** Spike-first, mirroring the proven image loop. Phase 1 (Task 1) discovers Flow's video UI by animating one real Karen slide end-to-end and records the recipe in `docs/superpowers/flow-video.md`. Phase 2 (Tasks 2–3) codifies that recipe into a standalone `animate-slide` skill and makes it discoverable. Storage/wiring follows Karen's existing path exactly: clip → `comics-v2/<comic>/anim/<key>/video.mp4` in the GCS bucket → `badcode assets-build` (renditions/poster/manifest) → `<AnimationWidget animation={comic.resolveAnimation('anim/<key>')} />`.

**Tech Stack:** Playwright MCP over CDP to a logged-in Chromium (`@playwright/mcp`), Google Flow (Veo image→video), `gsutil` (GCS), `@badcode/cli` `assets-build` (ffmpeg/sharp rendition+poster pipeline, already built), `@badcode/comic` (`AnimationWidget`/`VideoSource`, already built), Markdown skills.

## Global Constraints

- **No code changes to `@badcode/comic-meta` (no `comic.meta.ts` / schema change).** Source of truth for an animated panel is the storyboard record + the manifest. (Brainstorm decision: YAGNI.)
- **No new `@badcode/cli` code.** `gsutil` + the existing `badcode assets-build` cover upload and rendition/poster generation. Nothing to add or unit-test in the CLI.
- **Animation model: support both** — single source image + motion prompt (common), and an optional `to` end-keyframe tween.
- **Bucket pipeline only.** `animate-slide` targets a comic whose `assets.manifest.json` has `basePath: comics-v2/<comic>`. Local/v1 (`public/`, `baseUrl: ''`) comics are out of scope (must be migrated first).
- **Storage path convention (from Karen):** a clip lives at `comics-v2/<comic>/anim/<key>/video.mp4`; `assets-build` groups any folder containing a file named exactly `video.mp4` into an animation keyed by that folder (`anim/<key>`).
- **Voice:** all motion prompts and copy match `docs/voice.md` (sarcastic, dark, total authority; restraint over spectacle).
- **`badcode` CLI invocation:** `npm run badcode --workspace @badcode/cli -- <args>` (from repo root).
- **Flow completion is polled on a DOM/network signal, never a fixed sleep.**
- **Selectors are located by ARIA role + accessible name/placeholder, not by `browser_snapshot` refs (refs go stale).**
- This is a spike + documentation deliverable: there is **no unit-testable code**. "Tests" in this plan are concrete verifications — a clip that renders in the browser, `npm run typecheck` passing, and the skill reproducing the result on a second slide.

- [ ] **Step 5: Harvest the .mp4 via the signed-URL path**

Confirm the generated **video** media resolves like images do. In one `browser_run_code_unsafe` call resolve the signed URL (the sandbox has only `page` — use `page.evaluate` for the DOM and `page.request` for the fetch), mirroring the image recipe in `docs/superpowers/flow-selectors.md`:

```js
async (page) => {
  const name = await page.evaluate(() => {
    const els = [...document.querySelectorAll('video, img')]
      .map(e => e.currentSrc || e.src || '')
      .filter(s => s.includes('getMediaUrlRedirect'));
    return els.length ? new URL(els[els.length - 1]).searchParams.get('name') : null;
  });
  const resp = await page.request.get(
    'https://labs.google/fx/api/trpc/media.getMediaUrlRedirect?name=' + name);
  return { name, url: resp.url() };
}
```

Then download with the shell and confirm it is an MP4:

```bash
curl -sS "<signed-url>" -o /tmp/animate-spike/clip.mp4 && file /tmp/animate-spike/clip.mp4
```

Expected: `ISO Media, MP4 ...`. Record the harvest steps (and whether the `video`/`img` source selector differs from images) in `flow-video.md`.

- [ ] **Step 5: Write the gated per-slide loop**

Write the loop as **discuss → approve → produce** (mirror make-comic's gating). Steps, with the exact commands from Task 1:
1. Resolve the panel → manifest `basePath`, the current static image key (e.g. `img/i05`), the storyboard record `docs/<story>/storyboard/pNN.md`, and the new `anim/<key>`. Optional `to` panel for a tween.
2. Stage the source image: `gsutil cp gs://badcode-storage/comics-v2/<comic>/<imgKey>.<ext> <local>` (both images for a `to` tween).
3. **Gate:** discuss + approve the motion prompt before any generation.
4. Drive Flow per `docs/superpowers/flow-video.md` (source image + motion prompt + pinned aspect; poll completion).
5. Judge the clip (read poster / sampled frame); if weak, refine in the **same** Flow session and re-harvest.
6. Upload + build: `gsutil cp <clip>.mp4 gs://badcode-storage/comics-v2/<comic>/anim/<key>/video.mp4`, then `npm run badcode --workspace @badcode/cli -- assets-build -s comics-v2/<comic> -m apps/web/src/comics/<comic>/assets.manifest.json`.
7. Assemble: swap `<ImageWidget src={comic.resolve('<imgKey>')} />` → `<AnimationWidget animation={comic.resolveAnimation('anim/<key>')} />` (per AUTHORING.md). Verify `npm run typecheck` + spot-check in `npm run dev`.
8. Record (next step's format).

- [ ] **Step 6: Write the record format + resume + iterating**

Add the storyboard record format `docs/<story>/storyboard/pNN.md` (creating `storyboard/` if absent) capturing the **exact motion prompt**, Flow media id, `anim/<key>`, `model`, `status: planned|done`, and a **Revisions** log — mirroring make-comic's character/panel records. Add a **Resume** section (a `pNN.md` with `status: done` whose page already renders `AnimationWidget` is complete; else continue from the first incomplete step) and an **Iterating** section (*"just like that but slower"* — read the record, re-prompt Flow in the same session, re-harvest, re-`assets-build`, append a revision line; touch only that one panel).

- [ ] **Step 7: Verify the skill end-to-end on a SECOND slide**

The skill's "test" is reproduction. Follow the freshly-written `SKILL.md` literally to animate a **different** Karen slide (a second `anim/<key>`), with no improvisation beyond what the doc says. Expected: the loop runs to a rendered clip + a written `pNN.md` record. Fix any step in `SKILL.md` that was ambiguous or wrong; do not rely on memory of Task 1.
